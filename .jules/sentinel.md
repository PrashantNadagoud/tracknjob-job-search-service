## 2024-05-10 - Secure Error Logging in API

**Vulnerability:** The jobs API endpoints were using `print(f"Database error: {e}")` and `traceback.print_exc()` which print the raw exception message and the full stack trace of database errors to the standard output. This can potentially leak sensitive database structure information, table names, or internal stack details if the output is collected and inadvertently exposed (e.g., via a publicly accessible log aggregator or misconfigured logging endpoint). Furthermore, `print` blocks the event loop slightly, and `traceback.print_exc()` prints directly to `sys.stderr`, bypassing structured logging mechanisms.
**Learning:** Even if the direct HTTP response is secure (`detail="Database query failed"`), writing sensitive stack traces directly to `stdout`/`stderr` using `print_exc()` breaks structured logging conventions and violates defense in depth by exposing internal stack information to the environment logs unredacted. Standard `logger.error("...", exc_info=True)` should be used instead to allow the logging framework to handle, format, or redact stack traces if necessary.
**Prevention:** Always use proper logging frameworks (e.g., `logger.error("Message", exc_info=True)`) rather than raw `print()` and `traceback.print_exc()`. Avoid printing the raw error directly into string interpolations.
